## 1. What is the SiteFrame Payload?

The payload is a single text bundle (usually pasted into the WordPress admin â€œAI Bootstrapâ€ textarea) that describes *everything* needed to sync the theme: global CSS/JS chunks, header/footer snippets, and whole pages. SiteFrame parses the payload, updates ACF/global options, creates pages, and stores HTML/JS/CSS/JSON-LD into meta fields.

Everything is encoded using custom HTML comments:

```
<!--siteframe:part type="global_css"-->â€¦<!--/siteframe:part-->
```

and

```
<!--siteframe:page slug="..." title="..." ...-->
    <!--siteframe:block type="head"-->â€¦<!--/siteframe:block-->
    <!--siteframe:block type="html"-->â€¦<!--/siteframe:block-->
    â€¦
<!--/siteframe:page-->
```

This lets one big document represent all theme pieces.

## 2. `siteframe:part` Blocks (global assets/layout)

These correspond to theme-wide snippets stored via `siteframe_api_update_code_snippets()` / `siteframe_api_update_layout_snippets()`:

| Type               | Stored as                                     | Purpose                                                    |
|--------------------|-----------------------------------------------|------------------------------------------------------------|
| `global_css`       | `global_code_before_head_close`               | Injected before `</head>`                                 |
| `global_js`        | `global_code_before_body_close`               | Injected before `</body>`                                 |
| `global_header`    | `global_header_html` (layout snippet)         | Replaces header markup for all SiteFrame pages             |
| `global_footer`    | `global_footer_html` (layout snippet)         | Replaces footer markup                                     |
| Others (optional)  | e.g. `global_code_after_head_open`, etc.      | Any key accepted by `siteframe_apply_ai_parts()`           |

**Structure:**

```
<!--siteframe:part type="global_css"-->
â€¦CSSâ€¦
<!--/siteframe:part-->
```

Multiple parts allowed; later ones overwrite unless `action="append"`.

---

## 3. `siteframe:page` Blocks (individual pages)

Each page declaration creates/updates a WP page. Attributes:

- `slug`: desired slug (string)
- `title`: page title
- `name`: stable AI identifier (used for updates)
- `template`: e.g. `page-custom.php` (otherwise â€œdefaultâ€)
- `home="true"`: mark as homepage candidate
- Other flags (status, etc.) supported by parser.

**Inside each page:**

1. `<!--siteframe:block type="head"-->â€¦`  
   Extra `<head>` HTML (stored in `_custom_head_html`).

2. `<!--siteframe:block type="html"-->â€¦`  
   Main body HTML (stored in `_custom_raw_html`). Must include `<div data-siteframe-content>` around actual content.

3. Optional `siteframe:block type="css"`, `js`, `jsonld`, etc.  
   Stored in `_siteframe_ai_css`, `_siteframe_ai_js`, `_siteframe_ai_jsonld`.

Blocks are optional but recommended to keep order `head` â†’ `html` â†’ `css/js/jsonld`.

## 4. Placeholders within HTML

Shortcodes like `[ref]`, `[logo]`, `[image-1-url]`, `[pages-nav]`, etc., get resolved via `siteframe_field()` or WP shortcodes. Keep them intact.

Images inserted as `[image-1-url]` etc. correspond to global placeholder media. Literal `<img src="â€¦">` also fine.

## 5. How SiteFrame Processes Payload

1. Admin pastes payload into â€œAI payloadâ€ textarea and submits.
2. `siteframe_process_ai_payload()` parses text:
   - `siteframe_parse_ai_parts()` extracts global parts.
   - `siteframe_apply_ai_parts()` stores CSS/JS/header/footer.
   - `siteframe_parse_ai_payload()` iterates pages, calling `siteframe_insert_ai_page()`.
3. Each page:
   - Normalizes HTML (splits header/footer sections).
   - Stores CSS/JS/JSON-LD.
   - Sets template/meta fields.
   - Generates TOC anchors, WebP conversions, etc.
4. Home candidate flagged; if `home="true"` (or matches provided `home_identifier`), it can become the new front page.

---

## 6. Adding/Editing Pages

To modify a page:

- Locate its `siteframe:page` block via `name="..."` or `slug="..."`.
- Update inner blocks (head/html/css/js/jsonld).
- Keep `data-siteframe-content` wrapper.
- Ensure markup is valid; unescaped `--` inside comments must be avoided (standard HTML rules).

To add a page:

```
<!--siteframe:page slug="new-page" title="New Page" name="sf-new-page" template="page-custom.php"-->
    <!--siteframe:block type="head"-->â€¦<!--/siteframe:block-->
    <!--siteframe:block type="html"-->â€¦<!--/siteframe:block-->
    <!--siteframe:block type="css"-->â€¦<!--/siteframe:block-->
    <!--siteframe:block type="jsonld"-->â€¦<!--/siteframe:block-->
<!--/siteframe:page-->
```

`name` should be unique, lowercase, hyphenated (used as identifier). Template optional; if content includes complex layout/CSS, `page-custom.php` is recommended.

## 7. Decoding Payload Programmatically

A consuming AI should:

1. Split on `<!--siteframe:part` to collect parts (`type`, optional `action`, raw content).
2. Split on `<!--siteframe:page` to collect page definitions:
   - Parse attribute list (slug/title/name/template/home/etc.).
   - Within page block, find `<!--siteframe:block type="X"--> â€¦ <!--/siteframe:block-->`.
3. Output structured JSON (optional) mapping:
   ```

        {
          "slug":"strona-glowna",
   ```

4. When re-encoding, reverse process ensuring HTML comments remain verbatim.

## 8. Common Gotchas

- Nested `<!-- -->` inside block content can confuse parser; avoid or encode (e.g., use `&#60;!--` if necessary).
- JSON-LD blocks must remain valid JSON; watch for stray single quotes (convert to double).
- Keep `[table-of-contents]` placeholder if you need site-generated TOC; optimizer adds IDs automatically.
- Donâ€™t strip `<picture>` or `<source>`: sanitizer was updated to allow them.
- Last-modified headers rely on proper caching; no payload change needed, but be aware when testing.
